# Functional Requirements — ระบบ UBORROWU
**ระบบยืม-คืนพัสดุออนไลน์ มหาวิทยาลัยอุบลราชธานี**

> **Version:** 1.0  
> **วันที่:** กรกฎาคม 2569  
> **ขอบเขต:** ระบบยืม-คืนวัสดุและครุภัณฑ์สำหรับนักศึกษา เจ้าหน้าที่ และผู้ดูแลระบบ

---

## ภาพรวมผู้ใช้งาน (User Roles)

| บทบาท | คำอธิบาย |
|---|---|
| 🎓 **นักศึกษา (Student)** | ผู้ยืมสิ่งของจากระบบ มีสิทธิ์ยืม คืน และดูประวัติของตนเอง |
| 👷 **เจ้าหน้าที่ (Staff)** | ผู้อนุมัติการยืม-คืน และจัดการคลังสิ่งของ |
| 👑 **ผู้ดูแลระบบ (Admin)** | ผู้ควบคุมสิทธิ์ผู้ใช้งานทั้งหมดในระบบ |

---

## 1. ฟังก์ชันสำหรับนักศึกษา (Student Functional Requirements)

### 1.1 การแสดงและค้นหาสิ่งของ

| รหัส | Functional Requirement | Priority |
|---|---|---|
| FR-01 | ระบบต้องแสดงรายการวัสดุและครุภัณฑ์ทั้งหมด พร้อมรูปภาพ ชื่อสิ่งของ หมวดหมู่ สถานะ และจำนวนคงเหลือ | 🔴 High |
| FR-02 | ระบบต้องมีช่องค้นหา (Search) สิ่งของด้วยชื่อ และระบบกรองตามหมวดหมู่ ประเภท และสถานะ | 🔴 High |
| FR-03 | ระบบต้องแสดงหน้ารายละเอียดสิ่งของแต่ละชิ้น ได้แก่ ชื่อ รูปภาพ จำนวนคงเหลือ ตำแหน่งชั้นวาง และข้อมูลคิวรอ | 🔴 High |
| FR-04 | ระบบต้องแสดงหน้าแผนผังตำแหน่งชั้นวางของ (Position Map) จัดกลุ่มตามตู้หรือชั้นวาง | 🟡 Medium |

### 1.2 ระบบตะกร้า (Cart)

| รหัส | Functional Requirement | Priority |
|---|---|---|
| FR-05 | นักศึกษาต้องสามารถเพิ่มวัสดุหรือครุภัณฑ์ลงในตะกร้าได้ | 🔴 High |
| FR-06 | นักศึกษาต้องสามารถแก้ไขจำนวนสิ่งของในตะกร้าได้ | 🔴 High |
| FR-07 | นักศึกษาต้องสามารถลบสิ่งของออกจากตะกร้าได้ | 🔴 High |
| FR-08 | ระบบต้องลบตะกร้าที่ค้างเกินกำหนดเวลาโดยอัตโนมัติ | 🟡 Medium |

### 1.3 การยืมสิ่งของ (Borrowing)

| รหัส | Functional Requirement | Priority |
|---|---|---|
| FR-09 | นักศึกษาต้องสามารถกรอกแบบฟอร์มส่งคำขอยืมวัสดุ โดยระบุเหตุผลในการยืมได้ | 🔴 High |
| FR-10 | นักศึกษาต้องสามารถกรอกแบบฟอร์มส่งคำขอยืมครุภัณฑ์ โดยระบุเหตุผลและวันที่ต้องการใช้งาน | 🔴 High |
| FR-11 | ระบบต้องรองรับการส่งคำขอยืมสิ่งของหลายชิ้นพร้อมกันในคราวเดียว | 🔴 High |
| FR-12 | นักศึกษาต้องสามารถยกเลิกคำขอยืมที่ยังรอการอนุมัติอยู่ได้ | 🔴 High |
| FR-13 | นักศึกษาต้องสามารถกดยืนยันการรับสิ่งของ เมื่อเจ้าหน้าที่อนุมัติและจ่ายของให้แล้ว | 🔴 High |

### 1.4 การคืนสิ่งของ (Returning)

| รหัส | Functional Requirement | Priority |
|---|---|---|
| FR-14 | นักศึกษาต้องสามารถแจ้งคืนครุภัณฑ์ผ่านระบบเมื่อต้องการส่งของคืนได้ | 🔴 High |
| FR-15 | ระบบต้องแสดงสถานะการยืมปัจจุบัน เช่น รออนุมัติ / กำลังยืม / รอยืนยันการคืน ในหน้าหลัก | 🔴 High |

### 1.5 ระบบคิว (Queue)

| รหัส | Functional Requirement | Priority |
|---|---|---|
| FR-16 | เมื่อสิ่งของถูกยืมหมด นักศึกษาต้องสามารถกดจองคิวเพื่อรอยืมต่อได้ | 🔴 High |
| FR-17 | นักศึกษาต้องสามารถดูรายการคิวที่ตนเองจองอยู่ได้ | 🟡 Medium |
| FR-18 | นักศึกษาต้องสามารถออกจากคิวได้เมื่อไม่ต้องการยืมแล้ว | 🟡 Medium |
| FR-19 | ระบบต้องขยับคิวและแจ้งเตือนผู้จองคิวอันดับถัดไปโดยอัตโนมัติเมื่อของว่าง | 🔴 High |

### 1.6 ประวัติและการแนะนำ (History & Recommendation)

| รหัส | Functional Requirement | Priority |
|---|---|---|
| FR-20 | นักศึกษาต้องสามารถดูประวัติการยืมวัสดุและครุภัณฑ์ทั้งหมดของตนเองย้อนหลังได้ | 🔴 High |
| FR-21 | นักศึกษาต้องสามารถเสนอแนะสิ่งของใหม่ที่ต้องการให้จัดซื้อเข้าระบบ พร้อมระบุรายละเอียดได้ | 🟡 Medium |
| FR-22 | นักศึกษาต้องสามารถดูสถานะรายการที่ตนเองแนะนำว่าได้รับการอนุมัติแล้วหรือไม่ได้ | 🟡 Medium |

### 1.7 ข้อมูลส่วนตัว (Profile)

| รหัส | Functional Requirement | Priority |
|---|---|---|
| FR-23 | นักศึกษาต้องสามารถดูและแก้ไขข้อมูลส่วนตัวของตนเองได้ เช่น เบอร์โทรศัพท์ | 🟡 Medium |
| FR-24 | ระบบต้องบังคับให้นักศึกษากรอกข้อมูลติดต่อก่อนที่จะสามารถยืมสิ่งของได้ | 🔴 High |

## 2. ฟังก์ชันสำหรับเจ้าหน้าที่ (Staff Functional Requirements)

### 2.1 Dashboard และรายงาน

| รหัส | Functional Requirement | Priority |
|---|---|---|
| FR-25 | เจ้าหน้าที่ต้องสามารถเข้าถึงหน้า Dashboard ที่แสดง Summary Cards จำนวนการยืม-คืนปัจจุบัน | 🔴 High |
| FR-26 | ระบบต้องแสดงกราฟแท่ง (Bar Chart) ภาพรวมสถานะการยืมทั้งหมดบน Dashboard | 🟡 Medium |
| FR-27 | ระบบต้องแสดงกราฟโดนัท (Doughnut Chart) สัดส่วนจำนวนนักศึกษาปกติและที่ถูกระงับสิทธิ์ | 🟡 Medium |
| FR-28 | ระบบต้องแสดงรายการสิ่งของที่ถูกยืมมากที่สุด 3 อันดับแรก แยกระหว่างวัสดุและครุภัณฑ์ | 🟡 Medium |

### 2.2 การจัดการคำขอยืม-คืน

| รหัส | Functional Requirement | Priority |
|---|---|---|
| FR-29 | เจ้าหน้าที่ต้องสามารถดูรายการคำขอยืมวัสดุ/ครุภัณฑ์ที่รออนุมัติได้ทั้งหมด | 🔴 High |
| FR-30 | เจ้าหน้าที่ต้องสามารถกดอนุมัติ (Approve) คำขอยืมทีละรายการหรือหลายรายการพร้อมกันได้ | 🔴 High |
| FR-31 | เจ้าหน้าที่ต้องสามารถกดปฏิเสธ (Reject) คำขอยืมพร้อมระบุเหตุผลได้ | 🔴 High |
| FR-32 | เจ้าหน้าที่ต้องสามารถดูรายการครุภัณฑ์ที่กำลังถูกยืมอยู่ พร้อมวันครบกำหนดคืนได้ | 🔴 High |
| FR-33 | เจ้าหน้าที่ต้องสามารถกดยืนยันการรับคืนครุภัณฑ์ หรือปฏิเสธการคืนได้ | 🔴 High |
| FR-34 | เจ้าหน้าที่ต้องสามารถดูประวัติการยืม-คืนของนักศึกษาทุกคนในระบบได้ | 🔴 High |

### 2.3 การจัดการคลังสิ่งของ

| รหัส | Functional Requirement | Priority |
|---|---|---|
| FR-35 | เจ้าหน้าที่ต้องสามารถเพิ่มวัสดุใหม่เข้าระบบ พร้อมอัปโหลดรูปภาพ ระบุหมวดหมู่ และตำแหน่งวางได้ | 🔴 High |
| FR-36 | เจ้าหน้าที่ต้องสามารถแก้ไขข้อมูลวัสดุที่มีอยู่แล้วได้ | 🔴 High |
| FR-37 | เจ้าหน้าที่ต้องสามารถลบวัสดุออกจากระบบได้ทั้งแบบรายชิ้นและหลายชิ้นพร้อมกัน | 🔴 High |
| FR-38 | เจ้าหน้าที่ต้องสามารถเพิ่ม แก้ไข และลบครุภัณฑ์ รวมถึงกำหนดจำนวนวันสูงสุดที่อนุญาตให้ยืมได้ | 🔴 High |

### 2.4 การจัดการ Settings

| รหัส | Functional Requirement | Priority |
|---|---|---|
| FR-39 | เจ้าหน้าที่ต้องสามารถจัดการหมวดหมู่สิ่งของ (Category) ได้ เช่น เพิ่ม แก้ไข ลบ | 🟡 Medium |
| FR-40 | เจ้าหน้าที่ต้องสามารถจัดการสถานะสิ่งของ (Status) ได้ เช่น พร้อมยืม / ไม่พร้อมยืม | 🟡 Medium |
| FR-41 | เจ้าหน้าที่ต้องสามารถจัดการชื่อตำแหน่งตู้และชั้นวาง (Position) ในคลังได้ | 🟡 Medium |

### 2.5 การจัดการผู้ใช้และการระงับสิทธิ์

| รหัส | Functional Requirement | Priority |

| FR-43 | เจ้าหน้าที่ต้องสามารถระงับสิทธิ์ (Block) นักศึกษาที่ยืมของเกินกำหนดหรือทำของเสียหายได้ | 🔴 High |

| FR-45 | เจ้าหน้าที่ต้องสามารถปลดบล็อก (Unblock) นักศึกษาได้เมื่อนักศึกษาดำเนินการตามข้อตกลงครบถ้วน | 🔴 High |

| FR-47 | เจ้าหน้าที่ต้องสามารถกดอนุมัติหรือปฏิเสธรายการที่นักศึกษาแนะนำได้ | 🟡 Medium |

## 3. ฟังก์ชันสำหรับผู้ดูแลระบบ (Admin Functional Requirements)

| รหัส | Functional Requirement | Priority |
|---|---|---|
| FR-50 | แอดมินต้องสามารถดูรายชื่อผู้ใช้งานทั้งหมดในระบบ แยกตามบทบาท (นักศึกษา/เจ้าหน้าที่) | 🔴 High |
| FR-51 | แอดมินต้องสามารถอัปโหลดไฟล์ CSV เพื่อนำเข้ารายชื่อนักศึกษาที่มีสิทธิ์ใช้งาน (Whitelist) ได้ | 🔴 High |
| FR-52 | แอดมินต้องสามารถอัปโหลดไฟล์ CSV เพื่อนำเข้ารายชื่อเจ้าหน้าที่ที่มีสิทธิ์ใช้งาน (Staff Whitelist) ได้ | 🔴 High |
| FR-53 | แอดมินต้องสามารถดูรายละเอียดข้อมูลผู้ใช้งานแต่ละคนได้ | 🔴 High |
| FR-54 | แอดมินต้องสามารถลบผู้ใช้งานออกจากระบบได้อย่างถาวร | 🔴 High |
| FR-55 | แอดมินต้องสามารถระงับสิทธิ์ (Block) และปลดบล็อก (Unblock) ผู้ใช้งานได้ | 🔴 High |
| FR-56 | แอดมินต้องสามารถส่งออกไฟล์ CSV รายชื่อบุคลากรทั้งหมดในระบบได้ | 🟡 Medium |

## 4. ฟังก์ชันการทำงานของระบบ (System Automated Functions)

| รหัส | Functional Requirement | Priority |
|---|---|---|
| FR-57 | **ระบบยืนยันตัวตน (Authentication):** ระบบต้องรองรับการล็อกอินด้วยบัญชี Google OAuth 2.0 และตรวจสอบสิทธิ์จาก Whitelist ก่อนอนุญาตให้เข้าใช้งาน | 🔴 High |
| FR-58 | **ระบบบทบาทผู้ใช้ (Role-based Access):** ระบบต้องควบคุมการเข้าถึงหน้าต่าง ๆ ตามบทบาทของผู้ใช้ (นักศึกษา / เจ้าหน้าที่ / แอดมิน) | 🔴 High |
| FR-59 | **ระบบ AI แนะนำสิ่งของ (AI Recommendation):** ระบบต้องใช้ Apriori Algorithm วิเคราะห์ข้อมูลประวัติการยืม เพื่อแสดงรายการสิ่งของที่มักยืมคู่กันในหน้ารายละเอียดสินค้า | 🟡 Medium |
| FR-60 | **ระบบ QR Code:** ระบบต้องสามารถสร้างและแสดง QR Code สำหรับแต่ละรายการยืมได้ | 🟡 Medium |
| FR-61 | **ระบบ Scheduler:** ระบบต้องมีการลบตะกร้าสินค้าที่ค้างเกินเวลา และขยับคิวโดยอัตโนมัติ | 🟡 Medium |

## Priority Level

| ระดับ | ความหมาย |
|---|---|
| 🔴 **High** | ฟังก์ชันหลักที่ขาดไม่ได้ ระบบจะใช้งานไม่ได้หากไม่มีฟังก์ชันนี้ |
| 🟡 **Medium** | ฟังก์ชันเสริมที่ช่วยเพิ่มประสิทธิภาพ ระบบยังทำงานได้หากขาดไป |
| 🟢 **Low** | ฟังก์ชัน Nice-to-have ที่จะทำให้ระบบสมบูรณ์ขึ้น |
