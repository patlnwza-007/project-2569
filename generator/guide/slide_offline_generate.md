## 1. หลักการออกแบบโดยรวม (5 ข้อที่ห้ามผิด)

| หลักการ | ทำไมสำคัญ |
|---------|-----------|
| **1 สไลด์ = 1 ความคิดหลัก** | `<section>` ที่มี 2 หัวข้อในหน้าเดียวทำให้ผู้ฟังไม่รู้จะโฟกัสจุดไหน — 1 `<section>` ต่อ 1 ความคิดเสมอ |
| **ข้อความน้อยกว่าที่พูด** | สไลด์คือตัวช่วยความจำผู้ฟัง ไม่ใช่สคริปต์อ่าน — ถ้าอ่านสไลด์แล้วเข้าใจหมดโดยไม่ต้องฟัง แปลว่าเนื้อหาเยอะเกินไป |
| **ภาพแทนคำเมื่อทำได้** | สมองประมวลภาพเร็วกว่าตัวอักษร โดยเฉพาะ diagram/ตาราง/ไอคอน — ใช้ `<img>`/emoji/CSS shape แทนย่อหน้ายาว |
| **ทุกสไลด์ใช้ CSS class เดียวกัน** | กำหนด class ที่ `<style>` ใน `<head>` ครั้งเดียว (เช่น `.objective`, `.grid-3`, `.phase-card` — ดู §3.2) แล้วเรียกใช้ซ้ำทุกหน้า **ห้ามเขียน inline style ทับซ้ำแบบไม่มีมาตรฐาน** ไม่งั้นแต่ละหน้าจะหน้าตาไม่ตรงกัน |
| **ผู้ฟังต้องรู้เสมอว่าอยู่ตรงไหนของเรื่อง** | หน้า Outline + เลขหัวข้อกำกับทุกสไลด์ + progress bar/เลขหน้าที่ reveal.js ทำให้อัตโนมัติ (ตั้งค่าครั้งเดียว ดู §8.4) ทำให้กรรมการตามทัน จดอ้างอิง และย้อนถามได้แม่นยำ |

## 2. กฎความหนาแน่นของข้อความ (Text Density)

ใช้กฎ **5x5 แบบไทย** เหมือนกับฝั่ง Canva ทุกประการ (ตัวเลขไม่เปลี่ยนตามเครื่องมือสร้าง — คนอ่านคือคนเดิม):

- **ไม่เกิน 5 bullet ต่อสไลด์**
- **ไม่เกิน 1 บรรทัดต่อ bullet เมื่อฉายจริง และไม่เกิน 8–10 คำต่อ bullet** (นับคำภาษาไทยแบบคำต่อคำ) — ถ้ายาวกว่านี้ตอนแก้ไข ให้ตัดคำ ไม่ใช่ลด `font-size` ใน CSS
- **Title (h2) ไม่เกิน 6–8 คำ, Subtitle (`.objective`) ไม่เกิน 12–15 คำ**
- **รวมทั้งสไลด์ (ไม่นับ Title/Subtitle) ไม่ควรเกิน ~40 คำ** — ถ้าเกิน แปลว่า `<section>` นั้นมี 2 ความคิดซ้อนกัน ให้แตกเป็น `<section>` ใหม่แทนการยัดเพิ่ม
- **ไม่มีย่อหน้ายาว** — ถ้าต้องอธิบายเยอะ ให้ย้ายไปใน `<aside class="notes">` (ผู้ชมไม่เห็น โชว์เฉพาะ Speaker View — ดู §9)
- ถ้าเนื้อหาไม่พอในกฎนี้ → **แตกเป็น `<section>` ใหม่** (copy block เดิมในไฟล์ html แล้วแบ่งเนื้อหา) อย่ายัดใส่ `<section>` เดียวโดยลด `font-size`
- ระวังนิสัยของเบราว์เซอร์: reveal.js **ไม่หดฟอนต์อัตโนมัติ**เหมือน Canva — เนื้อหาล้นจะ**ล้นออกนอกจอจริงๆ** (ตัดขอบล่างหายไปเฉยๆ) ต้องเช็คด้วยตาทุกหน้าเสมอว่าเนื้อหาอยู่ในกรอบ ไม่ใช่รอให้ระบบเตือน

## 3. หลักการจัดวาง Layout และตำแหน่งอย่างมืออาชีพ

### 3.1 หลักการทั่วไปที่ใช้กับทุกสไลด์ (ดูก่อนเปิดตาราง §3.2)

ยึด 5 หลักการนี้เสมอ ไม่ว่า `<section>` นั้นจะใช้ layout อะไรในตาราง §3.2:

| หลักการ | ทำอย่างไรใน HTML/CSS |
|---------|----------------------|
| **Alignment (จัดชิดแนวเดียวกัน)** | ใช้ CSS Grid/Flexbox (`display:grid`/`display:flex`) แทนการกำหนดตำแหน่ง `position:absolute` มือ — เนื้อหาในคอลัมน์เดียวกันชิดแนวเดียวกันอัตโนมัติโดย browser ไม่ต้องวัดพิกเซลเอง |
| **Proximity (จัดกลุ่มสิ่งที่เกี่ยวข้อง)** | ใช้ `<div>` ครอบกลุ่มที่สัมพันธ์กัน + `gap`/`margin` ในกลุ่มเดียวกันให้น้อยกว่าระยะห่างระหว่างกลุ่ม (เช่น `.grid-3 { gap: 12px }` ระหว่างการ์ด แต่ `ul li { margin-bottom: 6px }` ภายในการ์ดเดียวกัน) |
| **Contrast / Hierarchy (ลำดับความสำคัญ)** | ใช้สเกล `font-size` ต่างระดับตาม §4 (Title > Subtitle > Body) + สีเน้น (`color`) ไม่เกิน 1 จุดต่อหน้า — อย่าใช้ `<strong>`/สีเน้นพร่ำเพรื่อจนไม่เหลืออะไรเด่น |
| **White space (พื้นที่ว่าง)** | เว้น `padding` ขอบ `<section>` อย่างน้อย 5–8% ของพื้นที่ (reveal.js ตั้งได้ที่ `Reveal.initialize({ margin: 0.05 })` ดู §8.2) และพื้นที่ที่มีเนื้อหาจริงไม่ควรเกิน ~70% ของหน้า |
| **Visual flow (ลำดับการมอง)** | คนอ่านตามแนว **Z-pattern**: ซ้ายบน (Title) → ขวาบน → ทแยงลงซ้ายล่าง → ขวาล่าง — จัด grid/flex ให้ลำดับ DOM ตรงกับลำดับที่อยากให้สายตาไหลผ่าน เพราะ browser จะ render ตามลำดับ element ในโค้ด |

### 3.2 Layout Pattern แนะนำต่อสไลด์ (โครงสร้าง HTML/CSS ต่อหมายเลขสไลด์)

> เลขสไลด์อ้างอิงตาม [`template_slides.md`](../template/template_slides.md) (9 หัวข้อหลัก + ปก + Outline)
> คอลัมน์ "โครงสร้าง HTML/CSS" อ้างอิง class จริงที่ใช้แล้วใน [`m9/slides-reveal.html`](../../m9/slides-reveal.html) — copy ชื่อ class เดิมไปใช้ได้ทันที ไม่ต้องคิด class ใหม่

| # | สไลด์ | Layout ที่แนะนำ | โครงสร้าง HTML/CSS | กฎเฉพาะ |
|---|-------|-----------------|---------------------|---------|
| 1 | หน้าปก | **Title Slide** เต็มจอสีพื้นเข้ม ตัวหนังสือขาวกึ่งกลาง | `<section data-background-color="{{สีหลัก}}">` + `<h1>` + `<p>` ชื่ออังกฤษ/ชื่อนักศึกษา | ห้ามใส่ bullet ใดๆ ในหน้านี้ |
| 1.5 | Outline | **Numbered list** อัตโนมัติด้วย CSS counter | `<ol class="toc-list" style="counter-reset:toc;"><li>...</li></ol>` (เลขวงกลมมาจาก `::before { content: counter(toc)". " }`) | แทรกหลังหน้าปก 1 หน้า — เลขหัวข้อต้องตรงกับหน้าชื่อทุกสไลด์ถัดไป |
| 2 | ที่มาและเหตุผล | **Title + Bullet 3 ชั้น** (บริบท → ปัญหา → ผลกระทบ) | `<ul>` ธรรมดา + `<strong>` เน้นคำเปิดแต่ละ bullet + `<div class="objective">` เป็น Subtitle | ใช้ลำดับเดียวกับ 3 ย่อหน้าใน proposal_ch1.md ข้อ 1.1 |
| 3 | Software Survey | **ตารางเปรียบเทียบ** สูงสุด 4 คอลัมน์ | `<table>` มาตรฐาน + คอลัมน์สุดท้าย `<th style="background:{{สีหลัก}}">` + เซลล์ `class="check"`/`class="cross"` (✅/❌) | ไฮไลต์คอลัมน์ระบบที่เสนอด้วยพื้นหลังต่างจากคอลัมน์อื่น |
| 4 | วัตถุประสงค์ | **Numbered List** | `<ol>` ธรรมดา (เลขอัตโนมัติจาก browser ไม่ต้องเขียนเลขเอง) | ไม่เกิน 4 ข้อต่อสไลด์ ถ้าเกินแตกเป็น `<section>` ใหม่ |
| 5 | ขอบเขตโครงงาน | **Multi-column card by Role** 3-4 คอลัมน์เท่ากัน | `<div class="grid-3">` หรือ `.grid-4` ครอบการ์ด role แต่ละใบ `<div style="background:{{สี role นี้}}">` + `<h4>` ชื่อ role + `<ul>` ความสามารถ | role เกิน 4 ให้แตกเป็น `<section>` ใหม่ ไม่บีบคอลัมน์แคบลง |
| 6 | เครื่องมือที่ใช้พัฒนา | **Icon/Card Grid** จัดกลุ่มตามหมวด (Frontend/Backend/DB/Infra) | `display:grid;grid-template-columns:1fr 1fr` การ์ดหมวดละใบ พื้นหลังสีอ่อนต่างกัน + emoji/โลโก้แทนไอคอน | ใช้โลโก้จริง (SVG/PNG ที่ vendor ไว้ local ตาม §8.1) แทนชื่อตัวหนังสือเมื่อทำได้ |
| 7 | System Overview | **Diagram-centric** ด้วยกล่อง+ข้อความเชื่อมโยง | `display:grid` การ์ดกล่องรอบ module หลัก + เส้น/ลูกศรวาดด้วย CSS border หรือแทรกภาพ SVG/PNG ที่ export จาก `architecture/overview.puml` | ถ้าวาด diagram ซับซ้อนเกิน CSS ให้ export ภาพจาก PlantUML มา `<img>` แทนดีกว่าฝืนวาดด้วย div |
| 8 | Functional Requirements | **ตารางสรุปตาม Epic** ไม่เกิน 6-7 แถว | `<table>` + `<p>` บรรทัดยอดรวมใต้ตาราง | Epic เกิน 7 ให้แตกเป็น `<section>` ใหม่ อย่าลด `font-size` ตารางจนอ่านไม่ออก |
| 9 | แผนการพัฒนา 3 เฟส | **3-Column Card** (หลัก \| รอง \| ที่เหลือ) | `<div class="grid-3">` ครอบ `.phase-card` 3 ใบ แต่ละใบ `style="border-top:4px solid {{สีเฟส}}"` (เขียว/เหลือง/ม่วงหรือแดง) | ใช้สีไล่ระดับความสำคัญให้สอดคล้องทั้ง 3 คอลัมน์ |
| 10 | Demo Mockup | **Image-heavy / Role Grid** | ถ้ามี screenshot จริง: `<img>` เต็มพื้นที่ + ตัวเลขวงกลม CSS `counter` กำกับลำดับ + `<p>` caption ใต้ภาพ · ถ้ายังไม่มี screenshot: การ์ด grid ต่อ role (เหมือน Slide 5) สรุปหน้าจอที่จะโชว์ | แตกเป็น 1 `<section>` ต่อ 1 role/flow ถ้ามีหลาย role — ห้ามปนหลาย flow ในหน้าเดียว |
| ปิดท้าย | Reference + Checklist | **สรุปปิด** รวมข้อมูลอ้างอิง + checklist ไว้หน้าเดียวกัน (แทน "ขอบคุณที่รับฟัง" เดี่ยวๆ) | `<table class="ref-table">` + `<div class="checklist-item">` ต่อบรรทัด (มี `::before{content:'☐'}`) | ใช้ปิด deck ให้ดูเป็นทางการและเช็คตัวเองก่อนสอบได้ในหน้าเดียว |

> **CSS class สำเร็จรูปที่ควรก็อปจาก `m9/slides-reveal.html`:** `.objective` (กล่อง Subtitle สีอ่อน), `.grid-3`/`.grid-4` (คอลัมน์เท่ากัน), `.phase-card` (การ์ดขอบบนมีสี), `.highlight-box` (กล่องสรุปเน้น), `.toc-list` (Outline เลขวงกลม), `.badge-p6`/`.badge-p7`/`.badge-p8` (ป้ายเฟสสี), `.check`/`.cross` (✅/❌ สีเขียว/แดง), `.footer-ref` (แหล่งอ้างอิงตัวเล็กมุมล่าง), `.checklist-item` (รายการ checklist มีกล่องหน้าอัตโนมัติ)

## 4. กฎด้านตัวอักษร (Typography)

reveal.js **สเกลทั้งสไลด์เป็นหน่วยเดียวกัน** ตามขนาดจอที่ฉายจริง (ตั้ง `width`/`height` ที่ config §8.2) — ต่างจาก Canva/PDF ที่ต้องคุมด้วย pt แบบสัมบูรณ์ สิ่งที่สำคัญคือ**สัดส่วนสัมพัทธ์ระหว่างองค์ประกอบ** ไม่ใช่ตัวเลข px/pt ตายตัว:

| ระดับ | ขนาดสัมพัทธ์แนะนำ (`em`) | ตัวอย่างการใช้ |
|-------|--------------------------|----------------|
| หัวข้อสไลด์ (h2 Title) | ใช้ค่า default ของ theme (`white`/`black`) ไม่ต้อง override เพิ่ม | ชื่อสไลด์บนสุด |
| เนื้อหาหลัก (bullet `<ul>`/`<ol>`) | `0.5–0.55em` | bullet, numbered list |
| ตาราง | `0.45em` | เนื้อตาราง, หัวตารางใช้ตัวหนาขนาดเดียวกัน |
| การ์ด/กล่อง (`.grid-3`, `.phase-card`) | `0.5em` container, หัวการ์ด (`h4`) `1em` ของ container | ขอบเขต, แผนพัฒนา |
| caption/label/แหล่งอ้างอิง | `0.35–0.4em` | `.footer-ref`, caption ใต้ภาพ demo |

**วิธีเช็คว่าเล็กเกินไปหรือไม่ (แทนเกณฑ์ pt ขั้นต่ำของ Canva):** กด **F11** (fullscreen) เปิด deck บนจอที่จะใช้ฉายจริง (หรือจอที่ใกล้เคียงที่สุด) แล้วถอยไปยืน/นั่งหลังห้องดูว่าอ่านตัวอักษรที่เล็กที่สุด (caption) ออกหรือไม่ — ถ้าอ่านไม่ออกให้เพิ่ม `em` ของ class นั้น ไม่ใช่ยัดเนื้อหาเพิ่มที่ขนาดเดิม

**ฟอนต์ที่แนะนำ (รองรับภาษาไทยครบ):**
- อันดับแรก: **`Sarabun`** — โครงสร้างเดียวกับ TH Sarabun New ที่ใช้ในเอกสาร proposal/fn req
- โหลด 2 วิธี (เลือกตามว่าต้องการ offline แท้ๆ แค่ไหน — ดู §8.1):
  - **ผ่าน CDN** (ง่ายสุด แต่ต้องมีเน็ตตอนเปิด): `<link href="https://fonts.googleapis.com/css2?family=Sarabun:wght@300;400;600;700&display=swap" rel="stylesheet">`
  - **Vendor ไฟล์ฟอนต์ไว้ในเครื่อง** (offline แท้ๆ — แนะนำสำหรับวันสอบจริง): ดาวน์โหลดไฟล์ `.woff2` ของ Sarabun จาก [Google Fonts](https://fonts.google.com/specimen/Sarabun) มาไว้ในโฟลเดอร์ `fonts/` แล้วประกาศเองด้วย `@font-face` ใน `<style>` (ตัวอย่างดู §8.1)
- **ห้ามใช้ฟอนต์ที่ไม่รองรับภาษาไทย** (ฟอนต์ละตินล้วน) — เบราว์เซอร์จะ fallback ไปฟอนต์ระบบเงียบๆ ทำให้สระบน/ล่างเยื้องผิดตำแหน่ง

**กฎการตัดบรรทัดภาษาไทย:** browser ตัดคำที่ช่องว่างเท่านั้น (เหมือน Canva) — ประโยคไทยยาวๆ ต้อง**เคาะวรรคเองตรงจุดที่ควรตัดคำ** หรือใช้ CSS `word-break: normal; overflow-wrap: break-word;` เป็นตาข่ายกันคำขาดกลางในกรณีจอแคบกว่าที่ออกแบบไว้

## 5. กฎด้านสี (Color System)

- **สีหลักไม่เกิน 2 สี + สีกลาง** (ขาว/เทา/ดำ) — ประกาศเป็น CSS custom property ที่ `:root` หรือ `.reveal` ครั้งเดียว เช่น `--primary: {{สีหลัก}};` แล้วเรียกใช้ `color: var(--primary)` ทุกที่ที่ต้องการ ไม่ hardcode hex ซ้ำหลายจุด
- **สัดส่วนการใช้สีแบบ 60-30-10:** พื้นหลัง/พื้นที่ว่างโทนกลาง (ขาว/เทาอ่อน) ~60% + สีรองสำหรับหัวข้อ/การ์ด/กราฟ ~30% + สีเน้น (accent) เฉพาะจุดสำคัญที่สุดของหน้านั้น ~10%
- **ความคมชัดของข้อความต้องอ่านออกจากที่นั่งหลังห้อง:** ใช้คู่สีตัวอักษร/พื้นหลังที่ contrast สูง — ถ้าไม่แน่ใจ เช็คด้วย contrast checker ตามเกณฑ์ WCAG ให้ได้อย่างน้อยระดับ AA (สำคัญกว่าปกติเพราะ browser ไม่ auto-adjust ให้เหมือน Canva theme)
- **ใช้สีสื่อความหมายให้สม่ำเสมอทุกสไลด์** สอดคล้องกับที่ใช้ในเอกสาร fn req/usecase:

| ความหมาย | สีแนะนำ (ตัวอย่างจริงจาก `m9/slides-reveal.html`) | ใช้ที่ไหน |
|----------|---------------------------------------------------|-----------|
| ทำงานเสร็จ/ใช้งานจริง | เขียว `#2e7d32` / `#43a047` | Slide 8-9 (ฟังก์ชันที่ active), `.check` |
| ยังไม่พัฒนา (dev in next version) | เทา `#888` | Slide 5, 8-9 (ฟังก์ชัน 🔜), `.footer-ref` |
| จุดเน้น/ระบบที่เสนอ | สีหลักของธีม (เข้ม) เช่น `#1a237e` | Slide 3 (คอลัมน์ระบบที่เสนอ), หัวตาราง `<th>` |
| คำเตือน/ข้อควรระวัง | ส้ม/เหลือง `#fb8c00` | จุดที่ซับซ้อน, `.badge-p7` |
| ผิด/ไม่รองรับ | แดง `#c62828` | `.cross` (❌) |

- **สีตาม role/actor (ถ้ามีหลาย role):** ให้แต่ละ role มีสีประจำตัวที่ใช้ซ้ำทุกจุดที่พูดถึง role นั้น (การ์ดขอบเขต Slide 5, การ์ด demo Slide 10) ผู้ฟังจะจำสีคู่กับ role ได้โดยไม่ต้องอ่านข้อความ
- หลีกเลี่ยงพื้นหลังไล่สี (gradient) เข้มเกินไปที่ทำให้ข้อความอ่านยาก โดยเฉพาะเมื่อฉายในห้องที่มีแสงสว่าง — พื้นขาว/เทาอ่อนเป็นฐานที่ปลอดภัยสุดสำหรับ reveal.js theme `white`

## 8. การสร้างสไลด์ด้วย reveal.js แบบ Offline (Content → HTML)

### 8.1 เตรียมไฟล์ให้เปิดได้โดยไม่ต้องต่ออินเทอร์เน็ต

ตัวอย่าง `m9/slides-reveal.html` ที่มีอยู่ในโปรเจกต์ใช้ CDN (`cdn.jsdelivr.net`, `fonts.googleapis.com`) ซึ่ง**สะดวกตอนแก้ไข**แต่**ต้องมีเน็ตตอนเปิด** — ถ้าต้องการ offline แท้ๆ (แนะนำก่อนวันสอบจริง) ให้ทำตามนี้:

1. สร้างโฟลเดอร์ `<module>/slides-reveal/` แล้ววาง `index.html` ไว้ข้างใน
2. ดาวน์โหลด reveal.js dist จาก [github.com/hakimel/reveal.js/releases](https://github.com/hakimel/reveal.js/releases) (หรือ `npm install reveal.js` แล้ว copy โฟลเดอร์ `node_modules/reveal.js/dist` และ `node_modules/reveal.js/plugin`) มาวางเป็น `<module>/slides-reveal/reveal.js/`
3. เปลี่ยน `<link>`/`<script>` ในไฟล์จาก CDN ให้ชี้ path local:
   ```html
   <link rel="stylesheet" href="reveal.js/dist/reveal.css">
   <link rel="stylesheet" href="reveal.js/dist/theme/white.css">
   <script src="reveal.js/dist/reveal.js"></script>
   ```
4. ดาวน์โหลดไฟล์ฟอนต์ `.woff2` ของ Sarabun จาก [Google Fonts](https://fonts.google.com/specimen/Sarabun) มาวางที่ `<module>/slides-reveal/fonts/` แล้วประกาศเองแทนลิงก์ Google Fonts:
   ```css
   @font-face {
     font-family: 'Sarabun';
     src: url('fonts/Sarabun-Regular.woff2') format('woff2');
     font-weight: 400;
   }
   @font-face {
     font-family: 'Sarabun';
     src: url('fonts/Sarabun-Bold.woff2') format('woff2');
     font-weight: 700;
   }
   ```
5. สร้างโฟลเดอร์ `images/` ข้างไฟล์ html สำหรับ screenshot mockup/diagram ทั้งหมด อ้างด้วย path สัมพัทธ์ `images/xxx.png` (ห้ามอ้าง path เต็มของเครื่อง)
6. ทดสอบ**ตัดอินเทอร์เน็ตแล้วเปิดไฟล์**ก่อนถือว่าเสร็จ — ถ้ายังมีอะไรค้าง (ฟอนต์ไม่ขึ้น/ไอคอนหาย) แปลว่ายังมีจุดที่พึ่ง CDN อยู่

### 8.2 ตั้งค่าทั้ง deck ครั้งเดียว — `<style>` ใน `<head>` (เทียบเท่า "หน้าแม่แบบ" ของ Canva)

ประกาศฟอนต์ + สีตาม Design System ที่กรอกไว้ใน `slide/<module>.md` **ครั้งเดียวที่ `<head>`** แล้วเรียกใช้ผ่าน class ทุกหน้า — ห้ามกำหนดสี/ฟอนต์แยกรายหน้า:

```html
<style>
  :root {
    --primary: {{สีหลัก เช่น #1A56DB}};
    --accent-1: {{สี role 1 ถ้ามี}};
    --accent-2: {{สี role 2 ถ้ามี}};
  }
  body, .reveal { font-family: 'Sarabun', sans-serif; }
  .reveal h1, .reveal h2, .reveal h3, .reveal h4 { color: var(--primary); }
  /* + class อื่นตาม §3.2 — copy จาก m9/slides-reveal.html แล้วแทนค่าสี */
</style>
```

ตั้งค่า `Reveal.initialize()` ท้ายไฟล์ครั้งเดียว (ควบคุมขนาดจอ/progress bar/เลขหน้าทั้ง deck):

```js
Reveal.initialize({
  controls: true,
  progress: true,       // progress bar อัตโนมัติ — ไม่ต้องวาดเอง
  slideNumber: 'c/t',   // เลขหน้าอัตโนมัติ มุมล่างขวา — ไม่ต้องทำ footer เอง
  hash: true,
  width: 1200,
  height: 700,
  margin: 0.05,          // = เว้นขอบ 5% ตามหลัก white space ใน §3.1
});
```

### 8.3 การ map เนื้อหาจากไฟล์รายละเอียดรายหน้า → HTML

แต่ละช่องในไฟล์ `slide/<module>.md` มีที่ลงตายตัวใน HTML — ไม่ต้องตีความใหม่:

| ช่องในไฟล์รายละเอียด | ไปอยู่ที่ไหนใน HTML |
|----------------------|----------------------|
| ชื่อหน้า เช่น "1 · ที่มาและเหตุผล" | `<h2>1 · ที่มาและเหตุผล</h2>` ต้นแต่ละ `<section>` |
| Subtitle (ประโยคใจความหลัก) | `<div class="objective">🎯 ...</div>` ใต้ `<h2>` |
| ข้อความที่ต้องพิมพ์ (bullet/ตาราง) | Bullet = `<ul>`/`<ol>` · ตาราง = `<table>` มาตรฐาน |
| Script พูด / Speaker Notes | `<aside class="notes">...</aside>` ท้าย `<section>` — โชว์เฉพาะ Speaker View (§9) ผู้ชมไม่เห็น |
| Layout ที่ระบุต่อหน้า | จัดตาม class ในตาราง §3.2 |
| หน้าที่ตัดเพิ่ม เช่น "(1/2)" | copy `<section>` เดิมทั้งก้อน → เปลี่ยนเลขใน `<h2>` เป็น (2/2) → เปลี่ยนเฉพาะเนื้อหา class/CSS คงเดิม |
| รูปภาพ (diagram/screenshot mockup/โลโก้) | `<img src="images/xxx.png">` — วางไฟล์ไว้ที่โฟลเดอร์ `images/` ตาม §8.1 ข้อ 5 |

### 8.4 องค์ประกอบนำทาง — Outline, เลขหัวข้อ, progress bar, เลขหน้า

reveal.js ทำ 2 ใน 4 กลไกให้อัตโนมัติจาก config เดียว (ต่างจาก Canva ที่ต้องทำ footer มือทุกหน้า):

1. **เลขหัวข้อ 1–9 ผูกกับหน้า Outline** — เขียนเองใน `<h2>` ทุกหน้าเนื้อหา (`2 · ซอฟต์แวร์...`) ให้ตรงกับ `<ol class="toc-list">` ที่หน้า Outline คำต่อคำ
2. **หน้า Outline** — 1 หน้าหลังปก ใช้ `<ol class="toc-list">` (ดู §3.2) พูด ~15 วินาทีตอนขึ้นหน้านี้
3. **Progress bar** — ตั้ง `progress: true` ครั้งเดียวใน `Reveal.initialize()` (§8.2) ได้เส้นแถบด้านล่างอัตโนมัติทุกหน้า ไม่ต้องวาดเอง
4. **เลขหน้า** — ตั้ง `slideNumber: 'c/t'` ครั้งเดียว ได้เลข "หน้าปัจจุบัน/ทั้งหมด" มุมล่างขวาอัตโนมัติทุกหน้า

### 10.1 ส่งไฟล์ให้กรรมการ/อาจารย์

- ถ้า vendor ไฟล์ตาม §8.1 ครบแล้ว: zip ทั้งโฟลเดอร์ `<module>/slides-reveal/` (มี `index.html` + `reveal.js/` + `fonts/` + `images/`) ส่งเป็นไฟล์เดียว เปิดได้จากเครื่องไหนก็ได้โดย double-click `index.html` ไม่ต้องติดตั้งอะไรเพิ่ม
- ทดสอบเปิดจากเครื่อง/บัญชีอื่น (หรือ USB flash drive) อย่างน้อย 1 เครื่องก่อนส่งจริง เพื่อยืนยันว่า path สัมพัทธ์ทำงานถูกต้องไม่ผูกกับเครื่องที่สร้าง

### 10.2 ข้อควรระวัง

- ถ้ายังใช้ CDN อยู่ (ยังไม่ได้ทำ §8.1) — deck จะ**เปิดไม่ได้/ฟอนต์หาย**ทันทีถ้าไม่มีเน็ต ห้ามพึ่งอินเทอร์เน็ตของสถานที่สอบ ให้ vendor ไฟล์ก่อนเสมอ
- Export PDF สำรอง (§9) ไว้เสมอ เผื่อเครื่องที่ใช้ฉายวันจริงเปิดเบราว์เซอร์ไม่ได้/เวอร์ชันเก่าเกินไป
- ภาพ mockup ที่แทรกเป็น `<img>` เป็น screenshot นิ่ง — ถ้าอยากให้กดลิงก์เปิด mockup จริงตอน demo สด ใส่ `<a href="...">` ทับภาพ แต่ต้องมีเน็ต/เปิด mockup server คู่กันตอนนั้น (คนละเรื่องกับตัวไฟล์สไลด์ที่ offline)

### 10.3 ส่งไฟล์เข้า repo (convention ของ repo นี้)

- ไฟล์รายละเอียดรายหน้า `slide/<module>.md` (เป็น source of truth ของเนื้อหา — แก้เนื้อหาให้แก้ที่นี่ก่อนแล้วค่อยไปแก้ใน HTML)
- ไฟล์ที่ vendor แล้วพร้อม offline เก็บที่ `<module>/slides-reveal/` ทั้งโฟลเดอร์ (หรือไฟล์เดี่ยว `<module>/slides-reveal.html` ถ้ายอมพึ่ง CDN ระหว่างพัฒนา แต่ก่อนสอบต้องแปลงเป็นแบบ vendor แล้ว)
- PDF ที่ export แล้ว วางคู่กันเช่น `<module>/slides-reveal.pdf`

- [ ] ทดสอบฉายจริงผ่านโปรเจกเตอร์/จอที่จะใช้จริงอย่างน้อย 1 ครั้ง (สีและฟอนต์อาจแสดงผลต่างจากหน้าจอที่แก้ไข)
- [ ] เปิดจากเครื่อง/USB อื่นอีก 1 เครื่อง ยืนยันว่า path สัมพัทธ์ทำงานถูกต้อง ไม่ผูกกับเครื่องที่สร้าง
- [ ] Export PDF สำรองแล้ว (`?print-pdf` + Print → Save as PDF) เปิดตรวจครบทุกหน้า (ฟอนต์ไทยไม่เพี้ยน รูปครบ) เก็บใส่เครื่อง/แฟลชไดรฟ์

- [ ] มีหน้า Outline หลังหน้าปก และเลขหัวข้อบนทุกสไลด์ตรงกับหน้า Outline คำต่อคำ (§8.4)
- [ ] Script พูดอยู่ใน `<aside class="notes">` ครบทุกหน้า (เช็คจาก Speaker View)
- [ ] มีสไลด์สำรอง (backup slides) ต่อท้าย deck สำหรับคำถามเชิงลึกที่กรรมการอาจถาม เช่น รายละเอียด FN เต็ม, use case diagram เต็ม — ไม่พูดถึงในเนื้อหาหลักแต่เปิดตอบได้ทันที
