# UBORROWU · ระบบยืม-คืนพัสดุออนไลน์ มหาวิทยาลัยอุบลราชธานี — Use Case Description

> อ้างอิงจาก `uborrowu_functional_requirements.md` (v1.2) — 1 use case ในเอกสารนี้ต้องตรงกับ 1 use case ใน `usecase.puml` (UC-01 ถึง UC-38)
> ผู้ใช้งาน 3 บทบาท: 🎓 นักศึกษา (Student), 👷 เจ้าหน้าที่ (Staff), 👑 ผู้ดูแลระบบ (Admin)
> จัดกลุ่มเป็น 10 Module (M1–M10) ตาม Django app หลังยุบรวม (ดูตารางสรุปท้ายเอกสาร) แต่ละ module แบ่งเป็น Epic (E1, E2...) ภายใน

**หมายเหตุทั่วไป (Cross-cutting concern):** ทุก use case ในเอกสารนี้ (ยกเว้น UC-16 การเข้าสู่ระบบเอง) มีเงื่อนไขก่อนเริ่มทำงานร่วมกันคือ **ผู้ใช้ต้องเข้าสู่ระบบด้วย Google OAuth 2.0 และผ่านการตรวจสอบสิทธิ์จาก Whitelist แล้ว (`login_required` + ตรวจสอบ role/permission ผ่าน django-allauth)** จึงไม่ระบุซ้ำในทุกแถว "เกี่ยวข้องกับ Service/Endpoint อื่น" ของแต่ละ use case ด้านล่าง

**หมายเหตุทั่วไป (การแจ้งเตือน — FR-39, FR-40, FR-41):** เหตุการณ์ต่อไปนี้ ระบบจะแจ้งเตือนนักศึกษาโดยอัตโนมัติเสมอทั้ง 2 ช่องทาง คือ (1) การแจ้งเตือนในหน้าเว็บ (in-app) และ (2) อีเมล — ผลการอนุมัติ/ปฏิเสธคำขอยืม (UC-18), ถึงคิวที่จองไว้ (UC-11), ใกล้ครบกำหนดคืน/เกินกำหนดคืนของครุภัณฑ์ (ต่อเนื่องจาก UC-08 จนกว่าจะแจ้งคืนสำเร็จที่ UC-09), คำขอถูกยกเลิกอัตโนมัติเพราะไม่มารับของภายใน 2 วัน (UC-18), และผลการพิจารณาคำเสนอแนะสิ่งของ (UC-29) — เป็นพฤติกรรมอัตโนมัติของระบบที่ผูกกับ use case ของ actor แต่ละอันตามที่ระบุ ไม่แยกเป็น use case ใหม่ เพราะไม่มี actor ที่ริเริ่มการแจ้งเตือนเองโดยตรง จึงระบุไว้ที่นี่ครั้งเดียวแทนการเขียนซ้ำในทุก use case ที่เกี่ยวข้อง

## M1 · account (บัญชีและการเข้าสู่ระบบ)

### E1 · การเข้าสู่ระบบ (ใช้ร่วมกันทุกบทบาท)

#### UC-16 · เข้าสู่ระบบด้วย Google OAuth

| หัวข้อ | รายละเอียด |
|-------|-----------|
| **รหัสอ้างอิง** | FR-17 |
| **Actor หลัก** | นักศึกษา (Student), เจ้าหน้าที่ (Staff), ผู้ดูแลระบบ (Admin) |
| **Actor รอง** | - |
| **คำอธิบายโดยย่อ** | ผู้ใช้ทุกบทบาทต้องการเข้าสู่ระบบด้วยบัญชี Google เพื่อเข้าใช้งานฟังก์ชันตามสิทธิ์ของตน โดยระบบตรวจสอบจากรายชื่อผู้มีสิทธิ์ใช้งาน (Whitelist) ก่อนอนุญาต และกำหนดบทบาทเริ่มต้นตามประเภท Whitelist ที่พบ เว้นแต่มีการกำหนดบทบาทเฉพาะบุคคล (Role Override) ไว้ล่วงหน้าโดยแอดมิน |
| **เงื่อนไขก่อนเริ่มทำงาน (Pre-condition)** | ผู้ใช้มีบัญชี Google ที่ถูกต้อง |
| **ลำดับเหตุการณ์หลัก (Basic Flow)** | 1. ผู้ใช้กดเข้าสู่ระบบด้วย Google<br>2. ระบบพาไปยืนยันตัวตนกับ Google OAuth 2.0<br>3. ระบบตรวจสอบอีเมลที่ยืนยันแล้วกับรายชื่อ Whitelist (นักศึกษา/เจ้าหน้าที่ — UC-37/UC-38) และกำหนดบทบาทเริ่มต้น (Student/Staff) ตามประเภท Whitelist ที่พบ<br>4. ระบบตรวจสอบว่าบัญชีนี้มีการกำหนดบทบาทเฉพาะบุคคล (Role Override, UC-34/UC-35) ไว้หรือไม่ — ถ้ามี ใช้บทบาทที่ถูกกำหนดแทนบทบาทเริ่มต้นจาก Whitelist<br>5. ระบบสร้างเซสชันผู้ใช้ตามบทบาทที่ได้ และพาเข้าสู่หน้าหลักของบทบาทนั้น |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | หากอีเมลไม่อยู่ใน Whitelist ระบบปฏิเสธการเข้าสู่ระบบและแจ้งเหตุผล<br>หากบัญชีถูกระงับสิทธิ์ (Block) โดยแอดมิน ระบบปฏิเสธการเข้าสู่ระบบ |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | ผู้ใช้มีเซสชันที่ยืนยันตัวตนแล้ว และเข้าใช้งานฟังก์ชันตามบทบาทที่แท้จริงของตนได้ (บทบาทจาก Whitelist หรือบทบาทที่ถูกกำหนดเฉพาะบุคคล) |
| **ความสัมพันธ์กับ Use Case อื่น** | เป็นเงื่อนไขก่อนเริ่มงาน (pre-condition) ร่วมของทุก use case อื่นในเอกสารนี้ |
| **กฎทางธุรกิจที่เกี่ยวข้อง (Business Rules)** | **ลำดับความสำคัญของบทบาท:** Role Override (UC-34/UC-35) มีผลเหนือกว่าบทบาทเริ่มต้นจาก Whitelist เสมอ ไม่ว่า Whitelist จะระบุเป็นประเภทใดก็ตาม — การตรวจสอบต้องทำตามลำดับ Whitelist ก่อน แล้วจึงเช็ค Override ทับ ไม่ใช่ตรวจพร้อมกัน |
| **เกี่ยวข้องกับ Service/Endpoint อื่น** | `account:login` (django-allauth Google provider) → `LoginView` (บริการภายนอกจริง: Google OAuth 2.0) |

### E2 · ข้อมูลส่วนตัว (Student)

#### UC-14 · จัดการข้อมูลส่วนตัว

| หัวข้อ | รายละเอียด |
|-------|-----------|
| **รหัสอ้างอิง** | FR-14, FR-15 |
| **Actor หลัก** | นักศึกษา (Student) |
| **Actor รอง** | - |
| **คำอธิบายโดยย่อ** | นักศึกษาต้องการดูและแก้ไขข้อมูลส่วนตัวของตนเอง เช่น เบอร์โทรศัพท์ เพื่อให้ข้อมูลติดต่อเป็นปัจจุบันและครบถ้วนสำหรับการยืมสิ่งของ พร้อมทั้งดูวันที่เข้าสู่ระบบครั้งแรก (First Login) และวันเวลาเข้าสู่ระบบครั้งล่าสุด (Last Login) ของตนเอง |
| **เงื่อนไขก่อนเริ่มทำงาน (Pre-condition)** | นักศึกษาเข้าสู่ระบบแล้ว |
| **ลำดับเหตุการณ์หลัก (Basic Flow)** | 1. นักศึกษาเปิดหน้าโปรไฟล์<br>2. ระบบแสดงข้อมูลส่วนตัว พร้อมวันที่เข้าสู่ระบบครั้งแรกและวันเวลาเข้าสู่ระบบครั้งล่าสุด (อ่านค่าจาก session/log ที่บันทึกไว้ตอน UC-16)<br>3. นักศึกษาแก้ไขข้อมูลติดต่อ เช่น เบอร์โทรศัพท์<br>4. ระบบบันทึกข้อมูลที่แก้ไข |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | หากข้อมูลติดต่อยังไม่ครบและนักศึกษาพยายามส่งคำขอยืม (UC-06) ระบบจะบังคับพากลับมาที่หน้านี้ก่อน (ตาม FR-15) |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | ข้อมูลส่วนตัวของนักศึกษาถูกอัปเดต และครบถ้วนพอสำหรับใช้ยืมสิ่งของ |
| **ความสัมพันธ์กับ Use Case อื่น** | เป็นเงื่อนไขก่อนเริ่มงาน (pre-condition) ของ UC-06 |
| **กฎทางธุรกิจที่เกี่ยวข้อง (Business Rules)** | - |
| **เกี่ยวข้องกับ Service/Endpoint อื่น** | `account:profile_detail`, `account:profile_update` → `ProfileDetailView`, `ProfileUpdateView` (อ่านค่า `date_joined`/`last_login` ที่บันทึกไว้ตอนเข้าสู่ระบบใน UC-16) |

| **คำอธิบายโดยย่อ** | แอดมินต้องการอัปโหลดไฟล์ CSV เพื่อกำหนดบทบาทเฉพาะบุคคล (Role Override) ให้ผู้ใช้หลายคนพร้อมกัน เช่น รายชื่อหัวหน้าห้องทั้งหมดในคราวเดียว แทนการกำหนดทีละคน |

| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | หากไฟล์มีรูปแบบไม่ถูกต้องหรือมีแถวข้อมูลผิดพลาด ระบบแจ้งรายการแถวที่ผิดพลาดและไม่นำเข้าแถวนั้น |

| **ความสัมพันธ์กับ Use Case อื่น** | `<<extend>>` ของ UC-34 (เป็นอีกวิธีหนึ่งของการกำหนดบทบาท ใช้เฉพาะเมื่อเลือกทำแบบกลุ่มผ่านไฟล์) |
| **กฎทางธุรกิจที่เกี่ยวข้อง (Business Rules)** | ใช้กฎเดียวกับ UC-34 (Role Override มีผลเหนือกว่า Whitelist เสมอ) กับทุกบัญชีในไฟล์ |

| **ลำดับเหตุการณ์หลัก (Basic Flow)** | 1. แอดมินเปิดหน้านำเข้ารายชื่อนักศึกษา<br>2. แอดมินอัปโหลดไฟล์ CSV รายชื่อนักศึกษา<br>3. ระบบตรวจสอบรูปแบบและเพิ่มรายชื่อในไฟล์เข้าสู่ Whitelist โดยไม่ลบรายชื่อเดิมที่มีอยู่ก่อนหน้า |
